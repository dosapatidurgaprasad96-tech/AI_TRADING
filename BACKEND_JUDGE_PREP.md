# Trade AI: Backend Technical Brief & Judge Preparation

This document provides a deep-dive into the backend architecture, data management, and AI integration of the **Trade AI System Engine**. Use this as a reference for Q&A sessions with judges.

---

## 🚀 1. Executive Summary
Trade AI is an enterprise-grade backend built on **Node.js** and **Express**, serving as the central "System Engine" for a high-performance trading ecosystem. It orchestrates real-time AI advisory nodes, secure portfolio management, and a proprietary allocation engine that bridges the gap between retail investors and professional trading expertise.

---

## 🛠️ 2. Technology Stack
- **Runtime:** Node.js (v20+)
- **Framework:** Express.js (RESTful Architecture)
- **Database:** MongoDB (Multi-collection segregation)
- **Object Modeling:** Mongoose
- **AI Integration:** OpenRouter SDK (NVIDIA Nemotron-3 Super 120B)
- **Authentication:** Google OAuth 2.0 + JWT + Bcrypt
- **Security:** Express-rate-limit, CORS, Helmet-style security headers

---

## 📊 3. Data Architecture & "Datasets"
While the system does not rely on static CSV files, it utilizes **Dynamic Live Datasets** generated by system interactions.

### A. Core Collections (The "Internal Dataset")
The system treats the following MongoDB collections as its primary data sources for AI context:
1.  **Customer Intelligence:** Risk appetite, market specialization, and portfolio complexity.
2.  **Trader Performance:** Experience levels, historical success rates, and specialization nodes.
3.  **Trade & Audit Logs:** Every trade execution and system change is logged for auditability (transparency/compliance).
4.  **Market Simulation:** Real-time generated ticks for BTC/USD, ETH/USD, AAPL, and TSLA, providing the environment for paper trading.

### B. AI Training Context
- **Primary Model:** `nvidia/nemotron-3-super-120b-a12b`.
- **Context Injection:** We use **RAG-style (Retrieval-Augmented Generation)** logic where the backend retrieves a user's "Client DNA" (risk/history) and feeds it into the LLM to generate personalized strategy reports.
- **Fail-safe Logic:** The system uses internal "Matching Templates" to ensure 100% uptime even if external AI APIs encounter latency.

---

## 🔒 4. Security & Compliance
- **Identity Isolation:** Separate schemas and middleware for Admins, Traders, and Customers.
- **JWT Integrity:** Tokens are cross-checked against collection identities to prevent role-escalation attacks.
- **Audit Trails:** A dedicated `AuditTrail` model tracks high-value operations (e.g., manual trader re-assignments).

## ❓ 5. Judge Q&A Preparation

### AI & Innovation
**Q: "What is the unique 'AI' value proposition here?"**
*   **A:** "It’s not just a chatbot. Our backend features an **Active Allocation Engine**. It uses a deterministic scoring algorithm to pair users, and *then* leverages an LLM to provide transparent, human-readable explanations for those technical pairings."

**Q: "How do you handle AI hallucinations in financial advice?"**
*   **A:** "We constrain the AI through **System Prompting**. The model is instructed to act as a 'System Auditor' or 'Financial Matching Node', limiting its output to Markdown-formatted strategy reports based strictly on the provided user metrics."

### Scalability & Infrastructure
**Q: "How would you handle a massive influx of users?"**
*   **A:** "The backend is stateless and horizontally scalable. By using MongoDB Atlas for the data layer and a stateless Express API, we can deploy multiple instances behind a load balancer without losing session integrity."

**Q: "Why separate the data into different collections?"**
*   **A:** "Security and performance. Physical segregation ensures that a leak in the 'Customer' node doesn't expose 'Admin' credentials, and it allows us to index each collection specifically for its use case (e.g., indexing Trader performance for fast matching)."
